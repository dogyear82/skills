# Question Strategy

Use this reference to decide what to ask next during requirements discovery.

## Goal

Ask the smallest set of questions that produces the largest increase in clarity.

## Prioritization Order

Prefer questions that unblock:

1. who the product is for
2. what problem it solves
3. what the user needs to do with it
4. what outcomes define success
5. what constraints limit the solution

## Question Rules

- Ask only a few questions per round.
- Prefer multiple-choice questions when they make the decision clearer.
- Use free-form questions when the answer space is unknown or genuinely open-ended.
- Avoid asking for details that do not change the product definition yet.
- Do not ask implementation questions unless they affect requirements directly.

## Good Question Types

- persona clarification
- workflow clarification
- priority clarification
- scope boundary clarification
- constraint clarification
- success criteria clarification

## Examples

Instead of:
- "Tell me everything you want this app to do."

Prefer:
- "Who is the primary user of this app?"
- "Which of these is the main job to be done: track work, automate a task, manage data, or communicate with others?"
- "Should this be optimized first for internal team use, public customers, or both?"

## Multiple-Choice Guidance

Use choices when:

- the likely options are known
- the decision has clear tradeoffs
- the user will benefit from seeing the shape of the decision

When providing options:

- keep the list short
- make options mutually understandable
- include a concise tradeoff when it matters

## Stop Condition Per Round

Stop the current round once the next most important uncertainty is isolated.
Then wait for the user's response before opening a new batch of questions.
