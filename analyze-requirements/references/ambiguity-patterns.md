# Ambiguity Patterns

Use this reference to inspect a request for missing or unclear requirements.

## Common Ambiguity Types

### Actor ambiguity

Unclear who the primary user, secondary user, admin, or external actor is.

### Problem ambiguity

Unclear what pain point or job to be done the product is solving.

### Workflow ambiguity

Unclear what sequence of actions the user must be able to perform.

### Scope ambiguity

Unclear what is in scope, out of scope, first-release scope, or future scope.

### Boundary ambiguity

Unclear where the system starts and stops, or what external systems it interacts with.

### Data ambiguity

Unclear what inputs, outputs, records, or artifacts the system must handle.

### Success ambiguity

Unclear how the user or business will know the product is successful.

### Constraint ambiguity

Unclear what limits exist around time, budget, technology, compliance, security, or operations.

## How to Use This Reference

During each discovery round:

- identify which ambiguity types are still open
- prioritize the ones blocking product definition most
- ask only the next most useful questions

## Output Discipline

Track each item as one of:

- confirmed
- assumed
- unknown

Do not let assumed items silently appear as confirmed facts.
